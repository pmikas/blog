Management Information System (MIS)

These notes are highly based on the course Introduction to Management Information Systems (MIS): A Survival Guide from edX, taught by Professor Alfonso Dur√°n Heras.

MISs:
  • Based on ICT (Information and Communication Tecnologies);
  • Examples: accounting systems, manufacturing planning systems;
  • Predetermined structure and scope of applicability;
  • Its design specifies which data will they use and how it will be processed;
  • Represent the application of ICT based solutions to specific business requirements;
  • There is a certain structure in how they combine and sequence the operations;
  • Support and structure information flows.

The tradicional development methodology "Waterfall" is divided in:
  1. Initial analysis
  2. Requirements definition
  3. Design
  4. Development
  5. Testing and acceptance
  6. Deployment
* It does not allow adding new functionality at a later stage, and the system should go through each phase once and only once.

Iterative methodologies: "Agile Manifesto"
  • Rapid prototyping with very short cycles;
  • Continuous delivery of usable software;
  • Encourages daily cooperation between business people and developers;
  • Welcomes changing requirements, even in late development.

Waterfall methodologies are more appropriate for critical and complex systems with very high quality requirements.

Involving adoption of technology-based solutions.

Network effects -> the benefits that each user gets, increase with the total number of users.
Moore's law -> increasing processing power per dollar.

Distributed IS: software and/or data is distributed over various computers, which
  • supports movements WITHIN the firm, NOT BETWEEN the firm and the customers;
  • both central computer and client computer can have software and data.

Per-to-per: commonly used for multimedia/data sharing, where each computer talks to the other computers directly.
Server-cliet: client nodes can run the software and request some data/services to the server when needed.

Distributed Infraestructure

Built on top of Telecommunication Networks and Computer Networks.

Standards:
  • Required in both wired and wireless networks;
  • May address different levels;
  • Crucial when different manufactures build products that must interoperate with each other.

LAN: Local Area Networks
WAN: Wide Area Networks
➥ Include LANs through appropriate networking gear (routers). ISP: Internet Service Providers

Cloud

Associated with distributing, computing and storage capacity over various locations accessible through the internet.
  • Private Cloud: all the resources are used and owned by the same company.
  • Public Cloud: one company sells service based on the resources to others (which use the resources). And allows dynamic allocation of resources.

Virtualization: set of tools and techniques that support the simulation of "virtul computers" based on the real physical computers.

Benefits of Private Clouds:
  • Load balancing;
  • Redundancy;
  • Standardization/Commoditization of the hardware.

Public and Private Clouds se the same basic "building blocks":
  • Several computer centers;
  • Software based virtualization tools;
  • Tools for dynamic allocation of resources.

Benefits of Public Clouds:
  • More sophisticated tools;
  • "Billing" capacity: pay per use.

On demand computing: allows to use only the resources needed at any point in time. You pay just for the resources you use.

Main types or levels of cloud computing services:
  • Infraestructure as a service
  • Software as a service

Centralized Information System: all the software and data of an IS reside in a central computer.

Software tools (ex: spreedsheet): programs that can manipulate any type of data in any structure.

Data and MIS

  • Data and its structure are a key componet of MIS;
  • A MIS is designed to support the information requirements and flows of business processes;
  • The data and its processing are defined during the MIS design.
Key components of MISs: data structure and programs.

Data
  • Used to refer to unprocessed, out-of-context set of numbers or text elements;
  • Is the IS raw material;
  • Differs from the processing of data.
* Once data has been processed and turned into information, it takes on context and structure. It becomes useful information, on which business decisions can be used.

Database: data + structure + definition + organization

Database Management System (DBMS)

Set of software programs specialized in managing databases.

Data modeling:
  • Process to logically structure or organize the data;
  • The resulting structure is referred as data model.
Hierarchical models:
  • Simple and efficient;
  • Inflexible;
  • There is a single path to a given data element.
Relational models:
  • Data is organized in tables;
  • Flexibility;
  • Each column contains the value of a given attribute for the individual represented in that row.
Database must store:
  • Values;
  • Structure: relationships and definition of data.
Foreign keys: relationships among tables.

Database Schema: is the data model described in the formal language of a DBMS.
➥ key component of a MIS together with its application progress.

Redundancy: when the same piece of data (or attribute) is stored in more than one place.

Inconsistency: when a attribute is stored in two different tables and have different information.
  • Different MISs might contain inconsistent data;
  • A given MIS might contain inconsistencies within its own database.
Normalization: its process leads to databases without redundancy.

DBMS:
  • Doesn't need to know the details of the stockroom;
  • Doesn't sets up a queue when there are several requests at the same time;
  • Doesn't validates and doesn't logs the requests;
  • Simplifies the development of the specialized application programs;
  • Doesn't need to worry about how the data is stored;
  • Carries out all the data management related tasks.

Data transactions
IS must guarantee:
  • All operations within a transaction are succesfully completed, or if something fails, all previous operations are cancelled;
  • If the DBMS, for whatever reason, doesn't get a valid "commit" from the application programs, it will automatically "roll back".
SQL: Structure Query Language
➥ Common DBMS language through which data can be defined, manipulated and queried.

Big Data Analytics: refers to the process of collecting, organizing and analyzing these "Big Data" sets to identify hidden patterns, previously unknown correlations, costumer preferences, market trends and other insights that can lead to better informed business decisions.

ERP: Enterprise Resource Planning
  • Integrated IS: encompasses a number of specialized modules (corporate functions);
  • Each module share the same data and database;
  • It can impose some constraints on how things are done.
Business Process: collection of related activities or tasks that produce a specific service or product, and thus create value for a costumer (internal or external).
➥ The tasks are grouped together because they are related and because they typically belong to a sequence.

Business Functions: encompasses all the tasks that are similar and therefore require the same skills or knowledge.

Business Transactions: too simple/small to be considered a process, it is the decoposition of business processes.
➥ Individual business transaction: each individual entity or object that goes through a business process.

Lifecycle of a business transaction: series of activities that make up the business process it must undergo.
  • Closed transactions: lifecycle completed
  • Open transactions: lifecycle not completed
Since business processes involve several business functions, data/information must flow from one business function to others. ERPs reduce this flow because the ERP's database is integrated with the modules of various business functions.

Each ERP module operate almost independently from the rest of the modules, except when they share a common database and DMS.
  • Allows each module to be developed, installed and evolved semi-independently from the other modules, avoinding complexity;
  • It also allows companies to install only some modules or to install them in several stages.
Sharing a database among several modules: all modules in a given ERP installation must use the same DBMS.
  • The logical structure or data model of database must cater for the needs of all ERP modules;
  • All ERP modules must be consistent with the logical structure of the database.
ERP's data module must encompass all the tables, attributes and relationships that any of the ERP modules might require; even if at that point in time, these specific tables are empty.

Development of the modules:
  • Can not be developed as fully independent projects;
  • Each module must comply with the data-model of the common database.
Implementation of the ERP:
  • Customer has to select modules that comply with database design;
  • All modules have to use the same DBMS.

ERP x Traditional MISs

PROS:
  • Support the integrated business processes encompassing multiple functions - eliminates redundancy and inconsistency;
  • Modularity: customers companies can choose which modules to install and might opt for a staged or phased approach.
CONS:
  • Complexity;
  • Limited choice and lock-in: "hostage" of the ERP provider.
"Best of Breed" approaches involve selecting the best individual solution for each of the company's needs.

Sometimes the company will adapt the ERP, or sometimes its processes. It the company adapts its processes, it does not necessarily have to adapt all of them, just those that are different from those implicit in the ERP. And it may adapt its processes to a "to be" state which is not the same as that initially implicit in the ERP.

Implicit Business Process Design: orchestrates how activities are carried out in a company through MIS (specially ERPs).

Business Process Reengineering (BPR)

Arguments for BPR:
  • MIS might enable or make advisable different process designs;
  • Takes advantages of "best practices";
  • Avoids the costs and issues involved in adapting the ERP;
  • Unsuitability of current practices.
Adaptation of the implicit business process design:
  • Reduce as much as possible the adaptation required - selection of an appropriate ERP package;
  • Requires an iterative approach, influenced by new options enable by ERP and best practices implicit in ERP.
Major ERPs are highly parameterized to built-in flexibility.

Coding-based Customization: computer programming to add additional programs and/or to modify the existing ones.
➥ Can lead to substantial costs, delays, risks and maintenance issues.

Modular approach: implementing initially just a subset of the modules we are planning to use.
Advantages:
  • Risk reduction;
  • Spread over a longer period of time: effort cost, business disruption.
Disvantages:
  • Requires intermediate setups: requires efforts, less-than-optimal support to the company.
Roll-out implentation: the system is implemented first in some locations the it is rolled-out gradually to the rest.
➥ Limited availability of support staff.

Key challenges in ERP implementation projects:
  • Selecting the appropriate ERP package;
  • Strinking balance between BPR and customization;
  • Preventing "customization creeping";
  • Avoid losing competitive advantages;
  • Overcoming business disruption;
  • Overcoming resistance to change;
  • Customization and post-implementation maintenance issues.

Transaction Processing System (TPS)

Structured decisions:
  • Alternatives predefined;
  • Logical decision process is clear;
  • The information is well specified;
  • Very frequent (repetitive);
  • Operational in nature;
  • Typically taken by operational personnel.
Unstructured decisions:
  • Alternatives not specified in advance;
  • Logical decision process is not predefined;
  • The information is not predetermined;
  • Typically one-off (each one is unique);
  • Tactical or strategic in nature;
  • Typically taken by managers (potentially supported by consultants or other highly qualified technical personnel).

DSS: Decision Support System
EIS: Executive Information System

TPS:
  • Data structure is designed in advance by their developers;
  • Data content is internally generated through the use of the TPS, and exchanged among TPSs.

Data warehouses (DW): relevant data from the database of the various transactional systems is periodically extracted and loaded into this DW.
➥ Main challenge: integration of the different data models.
